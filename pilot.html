Full password pilot

1 · Objective (pilot)

Design and implement an AI-driven Service Desk Automation (SDA) capability that:

  • Automates policy enforcement, decisioning, and orchestration
  • Provides context-aware guided workflows for end users
  • Reduces manual intervention for password reset scenarios
  • Ensures compliance with Microsoft identity and security policies

2 · Scope of work (pilot)

2.1 In-scope systems

Identity providers: Active Directory (on-premises), Azure Active Directory.

Endpoint environment: Domain-joined Windows devices.

Network dependency: VPN infrastructure.

ITSM platform: ManageEngine ServiceDesk Plus.

2.2 Use case coverage

  • Password reset requests
  • Account lockout resolution
  • Password sync issues (guided)
  • Remote vs office vs VM user handling

3 · Solution overview (pilot)

Decision & policy (automated): Query identity from AD / Azure AD; enforce minimum password age (24h), account status validation, and reset eligibility.

Outputs: Eligible / not eligible; user type classification; required flow path.

Orchestration (automated): Trigger reset via AD / Azure AD APIs; generate temporary password; conditional flags (e.g. force change at next login for office users); unlock account if required; update Service Desk ticket.

Guided resolution (human-in-the-loop): Deliver dynamic instructions from user context; capture confirmations such as “Login successful?” and “VPN connected?”; branch flows from responses. Channels may include ManageEngine Requests and MS Teams during the pilot only—post-pilot, ongoing automation is through ServiceDesk Plus, not Teams.

4 · Functional requirements (pilot)

4.1 Identity & policy validation

  • Integrate with AD / Azure AD
  • Retrieve account status and last password change timestamp
  • Enforce 24-hour minimum password policy

4.2 Password reset automation

  • Secure execution via service account
  • Generate temporary password
  • Apply conditional flags: office users → force change; VM users → no forced change

4.3 User classification logic

  • Remote vs office: IP, last login, device signals
  • VM users: Group membership

4.4 Guided workflow engine

  • Dynamic instruction generation
  • Branching on VPN availability, login success or failure
  • Automated request handling through (ManageEngine Requests; MS Teams only within pilot scope)

4.5 Exception handling

  • Password changed < 24h → reject with explanation
  • VPN failure → escalate
  • Login failure → guided retry or escalation
  • System unavailable → fallback to manual handling

4.6 Ticketing integration

  • Auto-update ticket: status, actions taken, timestamps
  • Maintain audit trail

  • Secure credential handling

Auditability

6 · Out of scope (pilot)

  • Full device control (e.g. forcing user login, restarting machines)
  • VPN infrastructure management
  • Endpoint-level automation requiring local admin interaction
  • Bypassing identity verification mechanisms

  • ITSM integration (ServiceDesk Plus APIs)

Full access request pilot

1 · Objective

Design and implement an automated access request workflow that:

  • Enforces role-based access control (RBAC) policies
  • Automates access provisioning decisions and execution
  • Eliminates invalid requests early (e.g., non-PM requests)
  • Minimizes human involvement to approval-only scenarios
  • Ensures auditability and compliance

2 · Scope of work

2.1 In-scope systems

ITSM platform: ManageEngine ServiceDesk Plus.

Identity / source of truth: Active Directory / Azure Active Directory.

Target system: File system / folder access control (e.g., shared drives, SharePoint, etc.).

2.2 Use case coverage

  • Access request submission
  • Role validation (PM vs non-PM)
  • Automated approval / rejection
  • Access provisioning
  • Exception handling (approval flows, invalid requests)

3 · Solution overview

3.1 Decision engine (fully automated)

Trigger: Executed automatically on request submission.

Responsibilities — validate request: requester identity; requested resource (Folder A).

Query system of record: user role (from AD / Azure AD or System X).

Evaluate policy: only PMs can request access.

Outputs: Approved / Rejected / Requires approval; access eligibility decision; validation flags.

3.2 Orchestration engine (automated execution)

  • If approved: grant access to Folder A.
  • If rejected: send system-generated response: “Please ask your PM to submit the request.”
  • If approval required: route to approver.
  • Update ticket: status; actions taken; decision outcome.

3.3 Guided resolution layer (minimal / conditional)

  • Only active when: approval is required; data is incomplete or invalid.
  • Behavior: no conversational prompts.
  • System-driven routing: approval workflow; request correction.

4 · Functional requirements

4.1 Request validation

  • Validate requester identity and resource (Folder A exists)
  • Reject invalid requests: missing or incorrect data

4.2 Role verification

  • Query AD / Azure AD / System X
  • Determine whether requester has PM role

4.3 Policy enforcement

  • If requester is PM: proceed with access logic
  • If requester is not PM: auto-reject request

4.4 Access provisioning

  • Grant access via AD group membership or direct ACL assignment
  • Validate access not already granted

4.5 Duplicate / existing access check

  • If user already has access: auto-close request
  • Add system note: “Access already exists”

4.6 Approval workflow (conditional)

  • Trigger only if resource requires additional approval
  • Route to folder owner / manager

4.7 Exception handling

  • Role system unavailable: set status = Pending; notify support
  • Invalid folder: reject request
  • Provisioning failure: escalate

4.8 Ticket updates

  • Automatically update status (Approved / Rejected / Pending), actions taken, timestamps
  • Maintain audit trail

5 · Non-functional requirements

Security

  • Enforce RBAC strictly
  • Secure service account for provisioning

Reliability

  • Retry provisioning on failure

Auditability

  • Full logging of access changes

Performance

  • Near real-time decisioning

Compliance

  • Align with access governance policies

6 · Out of scope

  • Manual access provisioning outside system flow
  • Role definition management (handled externally)
  • Complex entitlement modeling (beyond Folder A scenario)
  • Endpoint-level permission troubleshooting

7 · Dependencies

  • AD / Azure AD integration
  • Role source (System X)
  • Access control system (file system / SharePoint)
  • ServiceDesk Plus API
  • Service account with permission to grant access

8 · Deliverables

  • Automated request validation workflow
  • Role verification integration
  • Access provisioning automation
  • Rejection and approval workflows
  • Ticket update automation
  • Audit logging system
  • Configuration for policies and thresholds
